pip · venv · .env · project structure · golden rule

ADK Python is a pure Python library — no proxy, no separate server process. The cardinal rule: all LLM traffic routes through the LiteLlm wrapper. Never pass a raw model string or import a provider SDK directly. This gives you provider-agnostic routing, cost tracking, fallbacks, and retries for free.

MODULE 01

Basic Simple Agent

Agent · LiteLlm · Runner · InMemorySessionService · run()

The repo's Module 01 is the smallest practical ADK setup: one Agent (ADK's alias for LlmAgent), one LiteLlm(...) model wrapper, one cached Runner, and one InMemorySessionService. The implemented path is intentionally simple: run_prompt(...) calls blocking runner.run(...) and then extracts the final assistant text from ADK events.

from functools import lru_cache
from google.adk.agents import Agent
from google.adk.models.lite_llm import LiteLlm
from google.adk.runners import Runner
from google.adk.sessions import InMemorySessionService
from google.genai import types

def create_agent(settings) -> Agent:
    return Agent(
        name="root_agent",
        description="A tiny Google ADK agent that calls a LiteLLM-compatible endpoint.",
        instruction=settings.agent_instruction,
        model=LiteLlm(
            model=settings.litellm_model,
            api_base=settings.api_base,
            api_key=settings.api_key,
            max_tokens=settings.max_tokens,
        ),
    )

@lru_cache(maxsize=1)
def build_runner():
    settings = get_settings()
    return Runner(
        agent=create_agent(settings),
        app_name=settings.app_name,
        session_service=InMemorySessionService(),
        auto_create_session=True,
    )

def run_prompt(prompt: str) -> str:
    runner = build_runner()
    message = types.Content(role="user", parts=[types.Part(text=prompt)])
    events = runner.run(user_id="demo-user", session_id="...", new_message=message)
    return extract_final_text(events)

MODULE 04

Streaming Agent

run_async · stream_prompt · blocking fallback · chunked text output

The implemented Module 04 keeps the same single-agent stack as Module 01, but adds an async stream_prompt(...) helper that loops over runner.run_async(...) and yields text chunks as they arrive. The same module also keeps a normal blocking run_prompt(...) path so non-streaming callers can still get one final string.

streaming_agent/main.py
PYTHON 
def run_prompt(prompt: str) -> str:
    runtime = build_runner()
    events = runtime.runner.run(...)
    return extract_final_text(events)

async def stream_prompt(prompt: str):
    runtime = build_runner()
    message = _user_message(prompt)

    async for event in runtime.runner.run_async(
        user_id="demo-user",
        session_id="...",
        new_message=message,
    ):
        if getattr(event, "author", None) == "user":
            continue
        if not event.content or not event.content.parts:
            continue
        for part in event.content.parts:
            chunk = getattr(part, "text", None) or ""
            if chunk:
                yield chunk

When should the LLM call a tool?
• Call a tool when the answer depends on fresh or external data (weather, stock price, DB lookup)
• Answer directly when the user asks for formatting, summarization, or explanation of data already present
• Prefer narrow tool responsibilities: one tool fetches data, another transforms or formats it
• Put the rule in the instruction so the model does not guess when a real lookup is required

MODULE 06

Custom Agent — BaseAgent Subclass

BaseAgent · _run_async_impl · InvocationContext · yield Event

When LlmAgent isn't enough, subclass BaseAgent and implement _run_async_impl. You control everything: when to call LLMs, when to use rule-based logic, when to delegate to child agents, when to stop. Use cases: complex state machines, hybrid rule-based + LLM logic, agents with internal routing that doesn't need an LLM call to decide.

Module 06 in this repo lives under custom_agent/. Register "key": "custom_agent" with "module": "custom_agent.main" and "supports_streaming": true in agents.json — the FastAPI registry and React chat pick it up automatically (same pattern as Modules 04–05). In the UI, choose Custom Agent (keyword router); blocking calls use POST /api/chat, token streaming uses POST /api/chat/stream.

The implementation uses KeywordRoutingAgent (BaseAgent) with two LlmAgent children. Both specialists use LiteLlm(...) built from simple_litellm_agent.config.get_settings() — the same LITELLM_* .env as the rest of this project. Routing uses a TECH_KEYWORDS frozenset (core terms plus DB/cloud tooling in the file); user_text_for_routing_parts(...) normalizes user text the same way in the router and in routing_banner_markdown(prompt), so the visible route line always matches the branch that ran.

_run_async_impl contract
• Receives InvocationContext (ctx) — contains user_content, session, state
• Must be an async generator — use yield to emit Events
• Delegate to child agents with async with Aclosing(child.run_async(ctx)) as agen: async for event in agen: yield event
• Declare child agents as Pydantic model fields (not in __init__)
• Chat visibility: custom_agent/main.py prepends routing_banner_markdown(prompt) to each reply (**Route:** Tech specialist … or General specialist …) so the Streamlit and React UIs show which child handled the turn
custom_agent/agent.py (structure)
PYTHON 
from google.adk.agents import BaseAgent, LlmAgent
from google.adk.agents.invocation_context import InvocationContext
from google.adk.utils.context_utils import Aclosing
from google.genai import types

TECH_KEYWORDS = frozenset({"python", "code", "api", "docker", "postgres", "redis"})  # + more in repo

def user_text_for_routing_parts(parts) -> str:
    if not parts: return ""
    return " ".join(p.text or "" for p in parts if hasattr(p, "text")).lower()

def routing_banner_markdown(prompt: str) -> str:
    # Same keyword logic as the router — prepended to assistant text in main.py
    ut = user_text_for_routing_parts([types.Part(text=prompt)])
    is_tech = any(kw in ut for kw in TECH_KEYWORDS)
    return ("**Route:** Tech specialist (`tech_specialist`) · keyword match\n\n"
            if is_tech else "**Route:** General specialist (`general_specialist`) · no tech keywords matched\n\n")

class KeywordRoutingAgent(BaseAgent):
    tech_agent: LlmAgent
    general_agent: LlmAgent

    async def _run_async_impl(self, ctx: InvocationContext):
        user_text = ""
        if ctx.user_content and ctx.user_content.parts:
            user_text = user_text_for_routing_parts(ctx.user_content.parts)
        is_tech = any(kw in user_text for kw in TECH_KEYWORDS)
        chosen = self.tech_agent if is_tech else self.general_agent
        async with Aclosing(chosen.run_async(ctx)) as agen:
            async for event in agen: yield event

A multi-agent system (MAS) is a network of cooperating agents. In this module we build a business banking overdraft approval pipeline using ADK's SequentialAgent. Three specialist LlmAgent children run in order; each writes its analysis to session.state via output_key, and the next agent reads those results through {state_key} interpolation in its instruction.

Why SequentialAgent? Each specialist must complete its analysis before the next agent can read the results from session.state. The pipeline guarantees ordering without any LLM-based routing — SequentialAgent itself uses no model, only its children do.

Live Pipeline Audit Trail — The React UI shows a collapsible timeline of every agent transition and tool invocation as it happens during streaming. This gives full observability into multi-agent pipelines without opening a debugger.

How it works (three layers):

  1. Python (main.py) — stream_prompt() ignores events with no content.parts before tracking agents (avoids empty shell events confusing attribution). It resolves the active sub-agent with event.branch (leaf name) when author is missing or user, and falls back to a tool-name→agent map so every tool row still maps to deposit_agent, bill_agent, or decision_agent. agent_start/agent_end and tool_call/tool_result sentinels carry JSON-serialized input and output summaries.
  2. API (api_app.py) — The NDJSON generator detects the \x00AUDIT: prefix and emits {"type":"audit", "event":"tool_call", ...} instead of a text delta.
  3. React (App.jsx) — Audit events are attached to the current assistant message. AuditTrail groups rows by evt.agent and shows each tool’s full JSON arguments and JSON output summary in scrollable blocks — updating live during streaming.

What the user sees:

  • Each agent step shows a running/done status indicator
  • Each tool row shows the function name plus Input (arguments) and Output (tool return) as formatted JSON
  • Output summaries are derived in Python from known keys per tool (e.g. deposit counts, balances, demo_expected_decision for the overdraft tool)

MODULE 08

Workflow Agents

SequentialAgent · LoopAgent · ParallelAgent · no-LLM orchestrators

ADK workflow agents are pure orchestrators — they use no LLM themselves. In this repo's Module 08, they drive a modern retail deposit platform use case: concurrent risk checks, iterative exception reconciliation, and a final product-offer decision.

SequentialAgent

  • Runs sub-agents in order (pipeline)
  • Output of step N → session.state → step N+1
  • Use for: end-to-end product/ops pipelines with state handoff
  • Steps share state via output_key

LoopAgent

  • Repeats sub-agents until escalate_to_parent() is called
  • Set max_iterations as a safety cap
  • Use for: operational retries, queue drains, iterative reconciliation
  • Agent escalates when goal is achieved

ParallelAgent

  • Runs all sub-agents concurrently (asyncio)
  • Results collected when all finish
  • Use for: fan-out checks (risk, compliance, pricing, scoring)
  • Each sub-agent gets its own output_key

Composition Pattern

  • Workflow agents nest inside each other
  • SequentialAgent can contain ParallelAgents
  • All sub-agent LLM calls still use LiteLlm
  • Workflow agents themselves have no model= field

Agent-to-Agent (A2A) Protocol

remote delegation · agent discovery · tasks & messages · complements MCP

Modules 02–03 and 07–08 showed multi-agent patterns inside one process — orchestrators, pipelines, and workflow agents sharing memory through ADK. Modules 10–11 added MCP, a standard way for an agent to call tools and context servers. Agent-to-Agent (A2A) is the next interoperability layer: a standard way for one agent (or product) to delegate work to another agent over the network. A practical banking scenario is CD laddering: a customer-facing savings agent can hand off ladder design and maturity handling to a remote fixed-income specialist agent that knows how to structure staggered CD terms, preserve periodic liquidity, and decide whether each rung should be rolled into a new long-term CD or returned as cash.

MCP (Modules 10–11)

  • Model connects to tool / resource servers
  • Great for APIs, DBs, retrieval, “bring your own tools”
  • Thinks in tools, prompts, and context chunks

A2A (this module)

  • Agent connects to another agent as a peer
  • Great for specialist fleets, cross-bank servicing, and remote fixed-income / CD-ladder handoffs
  • Thinks in tasks, status, artifacts, and streaming collaboration
ConceptRole
Agent CardMachine-readable discovery document (capabilities, supported ladder types, maturity cadences, grace-period behavior, auth, endpoints) so the client agent can choose the right remote CD-ladder specialist.
Task / message flowStructured exchange: submit savings amount, target horizon, liquidity needs, and rollover preference; receive progress updates; then return a ladder plan with staggered maturities, APY comparisons, and rollover guidance.
Trust boundaryEach agent is its own security and policy domain — useful when the remote institution owns rate-shopping logic across banks, brokered-CD inventory, or reinvestment workflows that should stay on its side.

How this fits ADK: Your local banking assistant can gather customer goals, emergency-fund constraints, available cash, and desired access frequency, then call a remote A2A specialist that builds the actual CD ladder. That remote agent can reason about common ladder patterns such as 1/2/3/4/5-year rungs, shorter mini-ladders, periodic maturity access, APY shopping across institutions, and whether the next rung should be reinvested or left in cash. The lesson goal is protocol literacy: design delegation, timeouts, retries, approvals, and fallbacks explicitly — the same discipline you already use for MCP subprocesses and Redis-backed tools.

PYTHON 
from datetime import datetime, timedelta
from google.adk.tools import ToolContext

_MOCK_TRANSACTIONS = {
    "CUST-3001": [{"week":"W10","category":"dining","amount":280},{"week":"W11","category":"dining","amount":340},{"week":"W12","category":"dining","amount":410}],
    "CUST-3002": [{"week":"W10","category":"dining","amount":220},{"week":"W11","category":"dining","amount":215},{"week":"W12","category":"dining","amount":230}],
    "CUST-3003": [{"week":"W10","category":"grocery","amount":190},{"week":"W11","category":"grocery","amount":260},{"week":"W12","category":"grocery","amount":330}],
}

def get_weekly_transactions(customer_id: str) -> dict:
    """Return the latest weekly spending snapshot for a customer."""
    rows = _MOCK_TRANSACTIONS.get(customer_id, [])
    return {"customer_id": customer_id, "latest": rows[-1] if rows else {}, "all_weeks": rows}

def append_spending_snapshot(
    customer_id: str, week: str, category: str, amount: float,
    tool_context: ToolContext   # ADK injects — LLM never sees this
) -> dict:
    """Append a weekly spending snapshot to session.state['spending_log']."""
    log = tool_context.state.get("spending_log", [])
    log.append({"week": week, "category": category,
                "amount": amount, "recorded_at": datetime.utcnow().isoformat()})
    tool_context.state["spending_log"] = log
    return {"appended": True, "log_length": len(log)}

def check_trend_and_suppression(customer_id: str, tool_context: ToolContext) -> dict:
    """Deterministic guard — never calls an LLM. Returns trend + suppress flags."""
    log = tool_context.state.get("spending_log", [])
    if len(log) < 3:
        return {"trend_detected": False, "reason": "insufficient data"}
    recent = log[-3:]
    if len({r["category"] for r in recent}) > 1:
        return {"trend_detected": False, "reason": "mixed categories"}
    amounts  = [r["amount"] for r in recent]
    rising   = amounts[0] < amounts[1] < amounts[2]
    pct      = round((amounts[2] - amounts[0]) / amounts[0] * 100, 1)
    category = recent[0]["category"]
    cutoff   = datetime.utcnow() - timedelta(days=30)
    suppressed = any(
        s["category"] == category and s["response"] == "declined"
        and datetime.fromisoformat(s["date"]) > cutoff
        for s in tool_context.state.get("suggestion_history", [])
    )
    return {"trend_detected": rising, "category": category,
            "pct_increase": pct, "suppressed": suppressed, "amounts": amounts}

def record_suggestion_response(
    category: str, response: str, tool_context: ToolContext
) -> dict:
    """Record customer's response to a suggestion. response: 'accepted'|'declined'|'not_now'"""
    history = tool_context.state.get("suggestion_history", [])
    history.append({"category": category, "response": response,
                    "date": datetime.utcnow().isoformat()})
    tool_context.state["suggestion_history"] = history
    return {"recorded": True}

from datetime import datetime, timedelta
from google.adk.tools import ToolContext

def record_decision(
    customer_id: str, decision: str, risk_score: float, tool_context: ToolContext
) -> dict:
    """Append pipeline outcome to session.state['risk_history']."""
    history = tool_context.state.get("risk_history", [])
    history.append({"date": datetime.utcnow().isoformat(),
                    "decision": decision, "risk_score": risk_score})
    tool_context.state["risk_history"] = history
    return {"recorded": True, "total_events": len(history)}

def compute_risk_tier(customer_id: str, tool_context: ToolContext) -> dict:
    """
    Deterministic tier — never calls an LLM.
    Rules: 3+ DENY/CONDITIONAL in 30d → high; 1-2 or avg_score > 0.65 → medium; else low.
    Writes risk_tier and tier_updated_at to session.state.
    """
    history = tool_context.state.get("risk_history", [])
    cutoff  = datetime.utcnow() - timedelta(days=30)
    recent  = [h for h in history if datetime.fromisoformat(h["date"]) > cutoff]
    high_events = sum(1 for r in recent if r["decision"] in ("DENY", "CONDITIONAL"))
    avg_score   = sum(r["risk_score"] for r in recent) / max(len(recent), 1)
    tier = "high" if high_events >= 3 else "medium" if high_events >= 1 or avg_score > 0.65 else "low"
    tool_context.state["risk_tier"]       = tier
    tool_context.state["tier_updated_at"] = datetime.utcnow().isoformat()
    return {"risk_tier": tier, "high_events": high_events, "avg_score": round(avg_score, 3)}

def risk_tier_preflight(callback_context: CallbackContext) -> None:

    before_agent_callback — fires on every profile_agent invocation.

@dataclass

class FeedbackStore:
    def __init__(self, db_path: str = "feedback.db"):
        self.conn = sqlite3.connect(db_path, check_same_thread=False)
        self.conn.execute("""
            CREATE TABLE IF NOT EXISTS feedback (
                decision_id TEXT PRIMARY KEY,
                agent_name  TEXT,
                action      TEXT,
                outcome     TEXT,
                score       REAL,
                source      TEXT,
                context     TEXT,
                ts          TEXT
            )""")
        self.conn.commit()

    def record(self, fb: FeedbackRecord) -> None:
        d = asdict(fb); d["context"] = json.dumps(d["context"])
        self.conn.execute("""INSERT OR REPLACE INTO feedback
            VALUES (:decision_id,:agent_name,:action,:outcome,:score,:source,:context,:ts)""", d)
        self.conn.commit()

    def avg_score(self, agent_name: str, action: str, window: int = 50) -> float:
        row = self.conn.execute("""SELECT AVG(score) FROM (
              SELECT score FROM feedback WHERE agent_name=? AND action=?
              ORDER BY ts DESC LIMIT ?)""", (agent_name, action, window)).fetchone()
        return row[0] or 0.0

from dataclasses import dataclass
from typing import Any, Callable
import asyncio

@dataclass
class EvalCase:
    name: str
    inputs: dict
    expected: Any
    tags: list[str] = None

class OutcomeEvaluator:
    def __init__(self, agent_fn: Callable, scorer: Callable):
        self.agent_fn = agent_fn
        self.scorer = scorer
        self.results: list[dict] = []

    async def run(self, cases: list[EvalCase]) -> dict:
        for case in cases:
            output = await self.agent_fn(case.inputs)
            score  = self.scorer(output, case.expected)
            self.results.append({"name": case.name, "score": score,
                                  "pass": score >= 0.8})
        passing = sum(1 for r in self.results if r["pass"])
        return {"total": len(self.results), "passing": passing,
                "pass_rate": passing / len(self.results) if self.results else 0}

    EvalCase("safe_balance_no_alert",

import asyncpg, json, litellm

class BehavioralMemory:
    def __init__(self, pool: asyncpg.Pool):
        self.pool = pool

    async def setup(self):
        async with self.pool.acquire() as conn:
            await conn.execute("CREATE EXTENSION IF NOT EXISTS vector")
            await conn.execute("""
                CREATE TABLE IF NOT EXISTS user_memory (
                    id         BIGSERIAL PRIMARY KEY,
                    user_id    TEXT NOT NULL,
                    category   TEXT NOT NULL,
                    content    TEXT NOT NULL,
                    metadata   JSONB DEFAULT '{}',
                    embedding  vector(1536),
                    created_at TIMESTAMPTZ DEFAULT NOW()
                )""")

    async def _embed(self, text: str) -> list[float]:
        resp = litellm.embedding(model="text-embedding-3-small", input=[text])
        return resp.data[0].embedding

    async def remember(self, user_id: str, category: str,
                         content: str, metadata: dict = None) -> None:
        vec = await self._embed(content)
        async with self.pool.acquire() as conn:
            await conn.execute("""
                INSERT INTO user_memory (user_id, category, content, metadata, embedding)
                VALUES ($1, $2, $3, $4, $5)""",
                user_id, category, content, json.dumps(metadata or {}), vec)

    async def recall(self, user_id: str, query: str, top_k: int = 5) -> list[dict]:
        vec = await self._embed(query)
        async with self.pool.acquire() as conn:
            rows = await conn.fetch("""
                SELECT category, content, metadata,
                       1 - (embedding <=> $1::vector) AS similarity
                FROM user_memory WHERE user_id = $2
                ORDER BY embedding <=> $1::vector LIMIT $3""",
                vec, user_id, top_k)
        return [dict(r) for r in rows]

LLM as Judge vs Guardrails

evaluation · scoring · rule-based guards · two-layer safety

When an agent produces output, something needs to evaluate it. Guardrails are deterministic rule checks. LLM-as-Judge is probabilistic semantic evaluation. You need both — they catch different failure modes.

Guardrails (deterministic)

  • Amount within allowed range?
  • Output contains required fields?
  • PII detected in response?
  • Action within user permissions?
  • Fast (microseconds), 100% consistent

LLM-as-Judge (semantic)

  • Is the tone appropriate?
  • Does the explanation make sense?
  • Is the recommendation genuinely helpful?
  • Could this be misinterpreted?
  • Slower (100–500ms), probabilistic
evaluation_pipeline.py — guardrails + LLM judge in sequence
PYTHON 
import litellm, re, json

MODEL = "anthropic/claude-sonnet-4-5"

class Guardrails:
    MAX_TRANSFER = 1000.0
    REQUIRED_KEYS = {"action", "amount", "rationale"}
    PII_PATTERN = re.compile(r"\d{4}[- ]?\d{4}[- ]?\d{4}[- ]?\d{4}")

    def check(self, output: dict) -> tuple[bool, str]:
        missing = self.REQUIRED_KEYS - output.keys()
        if missing: return False, f"Missing keys: {missing}"
        if output.get("amount", 0) > self.MAX_TRANSFER:
            return False, f"Amount exceeds max {self.MAX_TRANSFER}"
        if self.PII_PATTERN.search(json.dumps(output)):
            return False, "PII detected in output"
        return True, "ok"

async def llm_judge(agent_output: dict, context: dict) -> dict:
    prompt = f"""Evaluate this banking agent output.
CONTEXT: {json.dumps(context)}
OUTPUT:  {json.dumps(agent_output)}
Score helpfulness, clarity, safety (each 0.0-1.0).
Respond ONLY with JSON: {{"helpfulness":0.0,"clarity":0.0,"safety":0.0,"overall":0.0,"flag":null}}
Set "flag" to explanation if any score < 0.7."""
    resp = litellm.completion(model=MODEL,
        messages=[{"role": "user", "content": prompt}],
        response_format={"type": "json_object"}, temperature=0)
    return json.loads(resp.choices[0].message.content)

async def evaluate_output(output: dict, context: dict) -> dict:
    passed, reason = Guardrails().check(output)
    if not passed:
        return {"approved": False, "reason": reason, "layer": "guardrail"}
    scores = await llm_judge(output, context)
    return {"approved": scores["overall"] >= 0.75 and not scores["flag"],
            "scores": scores, "layer": "llm_judge"}

@dataclass
class DecisionConfig:
    risk_threshold: float = 0.75      # trigger alert above this score
    alert_lead_time_hours: float = 2.0
    auto_transfer_max: float = 100.0
    prefer_sms: bool = True

class DecisionOptimizer:
    LEARNING_RATE = 0.05

    def __init__(self, feedback_store, config_path: str = "config.json"):
        self.feedback = feedback_store
        self.config_path = config_path
        self.config = self._load()

    def _load(self):
        try:
            with open(self.config_path) as f: return DecisionConfig(**json.load(f))
        except FileNotFoundError: return DecisionConfig()

    def optimize(self) -> dict:
        sms_score  = self.feedback.avg_score("overdraft_agent", "send_sms_alert")
        xfer_score = self.feedback.avg_score("overdraft_agent", "auto_transfer")
        alert_score= self.feedback.avg_score("overdraft_agent", "send_alert")
        changes = {}

        # Alerts under-performing → raise threshold (be more selective)
        if alert_score < 0.5:
            old = self.config.risk_threshold
            self.config.risk_threshold = min(0.95, old + self.LEARNING_RATE)
            changes["risk_threshold"] = (old, self.config.risk_threshold)

        # Transfers outperform SMS → switch channel preference
        if xfer_score > sms_score + 0.15:
            self.config.prefer_sms = False
            changes["prefer_sms"] = (True, False)

        with open(self.config_path, "w") as f:
            json.dump(asdict(self.config), f, indent=2)
        return {"updated": bool(changes), "changes": changes}

Exploration vs Exploitation

epsilon-greedy · UCB · multi-armed bandit · AI decision theory

Every adaptive system faces a fundamental tension: exploit the best-known strategy (safe, predictable) or explore alternatives to find something better (risky, potentially superior). This is the exploration-exploitation tradeoff — directly applicable to agent decision-making.

Pure Exploitation — always pick the highest-scoring action → risks local optima.

Pure Exploration — try random actions → thrashes, never learns consistently.

Epsilon-Greedy — exploit most of the time (1-ε), explore randomly with probability ε. Simple and effective.

bandit_agent.py — epsilon-greedy and UCB action selection
PYTHON 
import random, math
from typing import Literal

ACTIONS = ["send_sms", "send_push", "auto_transfer", "no_action"]

class EpsilonGreedySelector:
    def __init__(self, feedback_store, epsilon: float = 0.10):
        self.feedback = feedback_store
        self.epsilon = epsilon

    def select(self, agent_name: str) -> str:
        if random.random() < self.epsilon:
            chosen = random.choice(ACTIONS)
            print(f"[EXPLORE] Trying: {chosen}")
            return chosen
        scores = {a: self.feedback.avg_score(agent_name, a) for a in ACTIONS}
        best = max(scores, key=scores.get)
        print(f"[EXPLOIT] Best: {best} (score={scores[best]:.2f})")
        return best

class UCBSelector:
    """UCB1 — explores actions with high uncertainty more often."""
    def __init__(self, feedback_store, c: float = 1.41):
        self.feedback = feedback_store
        self.c = c

    def _count(self, agent_name: str, action: str) -> int:
        row = self.feedback.conn.execute(
            "SELECT COUNT(*) FROM feedback WHERE agent_name=? AND action=?",
            (agent_name, action)).fetchone()
        return row[0] or 1

    def select(self, agent_name: str) -> str:
        total = sum(self._count(agent_name, a) for a in ACTIONS)
        ucb = {a: self.feedback.avg_score(agent_name, a) +
               self.c * math.sqrt(math.log(total) / self._count(agent_name, a))
               for a in ACTIONS}
        best = max(ucb, key=ucb.get)
        print(f"[UCB] Selected: {best} (ucb={ucb[best]:.3f})")
        return best
⚠️
Exploration in production banking systems Keep epsilon low (5–10%). Never explore on irreversible financial operations — only on communication channel choice, alert timing, or message phrasing. Gate exploration behind a feature flag.

log = logging.getLogger("agent.production")
MODEL = "anthropic/claude-sonnet-4-5"

class CircuitBreaker:
    def __init__(self, threshold: int = 5, reset_secs: int = 60):
        self.failures = 0; self.threshold = threshold
        self.reset_secs = reset_secs; self.opened_at = None

    def is_open(self) -> bool:
        if self.failures >= self.threshold:
            if time.time() - self.opened_at > self.reset_secs:
                self.failures = 0; return False
            return True
        return False

    def record_failure(self): self.failures += 1; self.opened_at = time.time()
    def record_success(self): self.failures = 0

class ProductionAgent:
    def __init__(self, agent: LlmAgent, timeout_secs: float = 10.0):
        self.agent = agent
        self.timeout = timeout_secs
        self.breaker = CircuitBreaker()

    async def decide(self, user_id: str, event: dict, fallback_fn=None) -> dict:
        if self.breaker.is_open():
            log.warning("Circuit open — using fallback")
            return fallback_fn(event) if fallback_fn else {"action": "no_action"}
        try:
            svc = InMemorySessionService()
            sess = await svc.create_session(app_name="prod", user_id=user_id)
            runner = Runner(agent=self.agent, app_name="prod", session_service=svc)
            msg = Content(role="user", parts=[Part(text=json.dumps(event))])
            output = None
            async with asyncio.timeout(self.timeout):
                async for ev in runner.run_async(
                        user_id=user_id, session_id=sess.id, new_message=msg):
                    if ev.is_final_response(): output = ev.content.parts[0].text
            self.breaker.record_success()
            return {"action": output}
        except Exception as e:
            self.breaker.record_failure()
            log.error(f"Agent failure: {e}")
            return fallback_fn(event) if fallback_fn else {"action": "no_action"}
